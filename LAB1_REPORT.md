# Лабораторна робота 1: Наведення ладу в репозиторії

## Мета роботи

Провести аудит та оптимізацію існуючого репозиторію, встановити базові правила роботи з системою контролю версій Git та підготувати проєкт до подальшої розробки.

---

## Виконання завдань

У ході виконання роботи було проведено повний аудит бакалаврського проєкту (Telegram-бот для навчального закладу). Було виконано наступні кроки:

### 1. Публічність репозиторію
Репозиторій налаштовано як публічний для відкритого доступу до коду та можливості демонстрації роботи.

### 2. Оптимізація структури
- Видалено зайві тимчасові файли та директорії (`__pycache__`, `*.pyc`, `venv_test/`)
- Видалено тестові артефакти збірки та файли розкладу (`*.db`, `*.ics`)
- Очищено вміст віртуальних середовищ

### 3. Конфігурація Git
Додано файл `.gitignore`, специфічний для екосистеми Python:
- Тимчасові файли Python (`__pycache__`, `*.pyc`)
- Віртуальні середовища (`venv/`, `venv_test/`)
- Файли конфігурації середовища (`.env`, `.env.local`)
- IDE-специфічні папки (`.vscode/`, `.idea/`)
- Логи та системні файли

### 4. Документація
Створено файл `README.md` з:
- Описом функціціональності бота
- Переліком технологій
- Інструкціями з запуску

### 5. Ліцензування
Додано файл `LICENSE` з ліцензією MIT для вільного використання коду.

### 6. Безпека
Перевірено відсутність конфігураційних файлів із паролями або особистими даними:
- Використано `.env.example` як шаблон замість реальних даних
- `.env` додано до `.gitignore`
- Чутливі API-ключі не комітятся

## Структура проєкту

Нижче наведено перелік основних папок та файлів проєкту з поясненням їхнього призначення:

| Назва файлу / папки | Призначення |
|---------------------|-------------|
| `.gitignore` | Файл інструкцій для Git щодо ігнорування тимчасових файлів, віртуальних середовищ та конфігурацій |
| `README.md` | Основний файл документації проєкту (опис, інструкції з запуску, функціонал) |
| `LICENSE` | Юридичний документ (MIT License), що визначає правила використання коду |
| `requirements.txt` | Перелік залежностей Python-проєкту для встановлення через pip |
| `requirements-test.txt` | Залежності для запуску тестів |
| `main.py` | Точка входу в застосунок, ініціалізація бота |
| `config/` | Директорія конфігураційних файлів (налаштування бота, API) |
| `handlers/` | Директорія обробників команд бота (admin, student, teacher, cabinet) |
| `services/` | Директорія з зовнішніми сервісами (SUMDU API, парсери, планувальник) |
| `database/` | Директорія моделей даних та роботи з SQLite |
| `tests/` | Тести проєкту (юніт-тести та інтеграційні) |
| `utils/` | Допоміжні утиліти та декоратори |
| `.env.example` | Шаблон змінних середовища для налаштування бота |
| `.github/` | Конфігурація GitHub Actions (CI/CD) |

### Детальний опис директорій

#### `handlers/`
Модулі обробки команд Telegram-бота:
- `admin_commands.py` - адміністративні команди
- `student_commands.py` - команди для студентів
- `teacher_commands.py` - команди для викладачів
- `cabinet.py` - робота з особистим кабінетом
- `communication.py` - комунікація з користувачами
- `ics_schedule.py` - робота з розкладом у форматі ICS

#### `services/`
Інтеграція з зовнішніми системами:
- `sumdu_api.py` - API взаємодії з системою університету
- `sumdu_cabinet.py` - робота з особистим кабінетом
- `ics_parser.py` - парсер файлів розкладу
- `scheduler.py` - планування автоматичних задач

#### `database/`
Робота з даними:
- `models.py` - SQLAlchemy-моделі для SQLite
- `init.py` - ініціалізація підключення до БД

#### `tests/`
Тестування проєкту:
- `test_models.py` - юніт-тести моделей
- `test_integration.py` - інтеграційні тести

## Висновки

Під час виконання лабораторної роботи я закріпив навички роботи з Git, навчився структурувати репозиторій згідно з професійними стандартами розробки. Було проведено:

1. **Аудит репозиторію** - виявлено та видалено непотрібні файли
2. **Налаштування .gitignore** - захист від коміту чутливих даних
3. **Документування** - створено README та LICENSE
4. **Безпека** - перевірено відсутність приватних даних

Проєкт готовий до подальшого масштабування та спільної розробки, дотримано стандарти Python-спільноти (PEP 8, структура пакунків).
